# 证证鸽技术选型表

> 版本：`v0.1`  
> 目的：基于《证证鸽技术复用清单》确定第一阶段的正式技术选型，作为开发实现基准。  
> 核心原则：`先打通闭环，再增强精度；先做受控系统，不做开放式机器人。`

## 1. 选型目标

证证鸽第一阶段要跑通的最小闭环是：

1. 用户在指定页面发现疑似侵权内容
2. 通过插件一键取证
3. 服务端补抓页面并生成证据包
4. 完成品牌词/图片近似比对与风险评级
5. 通过钉钉或邮箱推送案件摘要
6. 用户在工作台选择模板，生成文书初稿
7. 审核人复核后导出正式版本

因此，技术选型优先考虑：

- 工程落地速度
- 与 `Hermes Agent` 的协同能力
- 对插件、浏览器抓取、文书生成、通知的支持
- 对权限隔离、审核流、证据留痕的可控性

## 2. 总体选型结论

第一阶段推荐采用：

- 插件端：`Plasmo + React + TypeScript`
- 工作台前端：`Next.js + React + TypeScript + Tailwind CSS`
- 后端 API：`FastAPI`
- 工作流编排：`Hermes Agent`
- 页面抓取：`Playwright`
- 近似比对：`RapidFuzz + pypinyin + imagehash`
- 文书生成：`docxtpl + python-docx`
- 通知：`Apprise + DingtalkChatbot`
- 数据存储：`PostgreSQL`
- 证据文件存储：`MinIO` 或本地对象存储目录
- 缓存 / 任务协调：`Redis`

## 3. 分层技术选型表

| 层级 | 选型 | 角色 | 为什么选它 | 备选 / 不选原因 |
| --- | --- | --- | --- | --- |
| 浏览器插件 | `Plasmo` | 页面内一键取证、内容采集、提交工作台 | 插件工程化成熟，React/TS 友好，适合快速做 popup/content script/background | 原生 Manifest 从零开发成本高 |
| Web 工作台 | `Next.js` | 案件列表、证据预览、文书生成、审核流 UI | 路由、SSR/CSR 混合能力成熟，适合后台系统快速搭建 | 纯 Vite 也可做，但后台页面组织和后续扩展略弱 |
| 后端 API | `FastAPI` | 提供案件、证据、任务、模板、审核等业务接口 | Python 生态与比对、文书、通知库一致，开发效率高 | Node 生态也能做，但文书与比对链路会更分裂 |
| 编排中枢 | `Hermes Agent` | 后端工作流编排、监测调度、通知触发、文书生成任务编排 | 适合承接多步骤后台链路，是本项目后端的智能编排层 | 不适合作为直接暴露给用户的开放聊天入口 |
| 页面抓取 | `Playwright` | 补抓 HTML、全页截图、动态内容加载 | 对电商页、官网页兼容度高，现成稳定 | requests/bs4 对动态页面太弱 |
| 文本比对 | `RapidFuzz + pypinyin` | 商标近似、品牌词变体、标题比对 | 轻量、够快、对中文品牌场景实用 | 纯 embedding 第一版成本高、解释性弱 |
| 图片比对 | `imagehash` | logo、商品图快速预筛 | P0 足够轻量，容易落地 | CLIP / imagededup 放到增强版 |
| 文书生成 | `docxtpl + python-docx` | 律师函、投诉函、举报材料初稿导出 | 模板化稳定，适合法务场景 | 纯 LLM 自由生成不稳定 |
| 通知 | `Apprise + DingtalkChatbot` | 邮件通知、钉钉消息推送 | 邮件和钉钉都能快速落地，适合 P0 | 飞书、企微作为扩展保留 |
| 数据库 | `PostgreSQL` | 案件、证据元数据、模板、审核记录 | 关系数据和审计留痕适配度高 | SQLite 仅适合极简原型 |
| 对象存储 | `MinIO` | 截图、HTML、文书、证据包存储 | 与 S3 兼容，后续迁移方便 | 纯本地磁盘可开发期先用，但不适合长期方案 |
| 缓存 / 协调 | `Redis` | 任务状态、去重、短期缓存 | 和异步任务、通知、限流配合方便 | 无状态直跑会让任务管理混乱 |

## 4. Hermes Agent 的明确定位

`Hermes Agent` 在证证鸽中不作为前台聊天机器人，而作为后端编排中枢。

### 4.1 Hermes 负责什么

- 定时巡检任务编排
- 主动取证后的后台补抓与任务串联
- 调用比对工具并汇总风险评分
- 触发钉钉 / 邮件通知
- 驱动文书草稿生成
- 接收人工审核回流后的模板优化信号

### 4.2 Hermes 不负责什么

- 用户身份认证
- RBAC / ACL 权限控制
- 案件级数据可见性判断
- 直接对外暴露自由聊天能力
- 代替人工作出“侵权成立”结论

### 4.3 设计约束

所有用户请求先进入业务后端，由后端根据角色、案件权限、动作类型进行校验，再决定是否触发 Hermes。

推荐链路：

`插件 / 工作台 / 钉钉动作 -> FastAPI 权限网关 -> Hermes Agent -> 工具执行 -> 结果回写`

## 5. 第一阶段正式采纳能力

### 5.1 P0 直接开发

- 插件一键取证
- 工作台案件管理
- 服务端补抓页面
- 文本和图片近似预筛
- 风险分级
- 证据包生成
- 钉钉 / 邮箱通知
- 律师函与平台投诉函初稿生成
- 人工审核与导出

### 5.2 P1 预留接口，不在第一阶段强做

- 指定站点自动巡检
- 举报材料生成
- 多模板体系扩展
- 海外站点 / 多法域扩展
- 更强的语义与多模态比对

### 5.3 P2 只保留架构入口

- 飞书、企业微信、短信等新通知通道
- 历史案例驱动的受控学习
- 大规模多站点监控
- 与公证处 / 法院系统的深度集成

## 6. 当前明确不选的方案

以下方向不纳入第一阶段正式技术栈：

- `changedetection.io` 全量直接接入  
  原因：适合借鉴，不适合第一版直接做成核心依赖。

- `Crawl4AI` 作为主抓取层  
  原因：增强能力有价值，但第一版用 Playwright 更直接。

- `CLIP / sentence-transformers / imagededup` 作为 P0 主比对链  
  原因：能力更强，但会抬高调试成本和解释难度。

- `Hermes` 直连用户钉钉自由聊天  
  原因：权限边界不稳，容易把案件、模板和工具暴露过头。

## 7. 风险与补充说明

### 7.1 许可风险

- `docxtpl` 为 `LGPL-2.1`，正式商用前要再做一次许可证评估。

### 7.2 工程风险

- Playwright 抓取稳定性依赖页面结构，需做超时、重试、截图兜底。
- 钉钉机器人第一阶段建议先做“推送型 + 受限动作型”，不做开放问答型。
- 文书必须坚持“初稿 + 审核流”，不直接输出“正式法律结论”。

### 7.3 架构风险

- 如果把 Hermes 当成通用后台而没有外层权限网关，后续很容易失控。
- 取证数据和案件文书都带敏感性，必须先设计好角色与审计日志。

## 8. 推荐落地顺序

1. 先搭 `FastAPI + PostgreSQL + MinIO + Redis` 基础后端
2. 用 `Plasmo` 做插件取证入口
3. 用 `Playwright` 打通补抓、截图、HTML 保存
4. 落 `RapidFuzz + pypinyin + imagehash` 的风险预筛链
5. 接 `Apprise + DingtalkChatbot` 通知
6. 接 `docxtpl + python-docx` 文书初稿导出
7. 最后接 `Hermes Agent` 把各环节编排成完整工作流
