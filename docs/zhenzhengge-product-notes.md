# 证证鸽产品共识与关键决策

> 本文档用于沉淀前期讨论过程中已经达成的核心共识，避免后续在产品定位、功能边界、法律表述和技术实施上反复摇摆。

## 1. 项目名称

- 当前主品牌名：`证证鸽`
- 当前产品方向：`知识产权侵权取证与处置辅助平台`

## 2. 核心定位

证证鸽不是一个“自动判案工具”，也不是一个泛化法律问答机器人，而是一套面向知识产权侵权响应场景的智能工作流系统。  
它的核心价值是把分散、易失的线上侵权线索，快速转化为：

- 可留存的证据包
- 可管理的案件材料
- 可进入法务审核流程的文书初稿
- 可推送的风险预警事件

## 3. 最终想解决的问题

目标用户在面对知识产权侵权时，通常会遇到以下问题：

- 侵权页面随时可能消失，`证据易失`
- 发现线索后，`不会标准化取证`
- 即使截图了，也很难形成`可管理、可追溯的证据材料`
- 后续投诉、发函、举报或诉讼前，需要整理大量事实与模板，`法务准备成本高`
- 长期盯站、巡检、发现更新内容很耗人力，`监测成本高`

证证鸽的目标是解决上述几个前置痛点。

## 4. 产品重心

当前版本的重心明确放在：`取证固证`

后续的法律文书辅助、举报材料生成、投诉材料输出，都是围绕“取证固证后如何继续推进案件处置”的连带能力，用来形成完整闭环。

因此主次顺序应明确为：

1. 发现侵权线索
2. 标准化取证与固证
3. 风险分析与案件结构化
4. 文书辅助与审核流
5. 举报/投诉/维权准备

## 5. 两条核心使用链路

### 5.1 主动取证模式

适用于用户自己已经发现疑似侵权页面的场景。

流程如下：

1. 用户在网页中发现疑似侵权内容
2. 通过浏览器插件一键发起取证
3. 系统生成标准化证据包
4. 系统进行相似度比对与风险评级
5. 生成案件摘要、证据目录与建议动作
6. 用户可选择生成律师函、投诉函、举报材料等文书初稿
7. 进入人工审核与导出流程

### 5.2 自动监测模式

适用于用户希望平台持续盯站、自动发现新风险的场景。

流程如下：

1. 用户配置监控任务
2. 平台对指定站点/页面/栏目进行定时巡检
3. 页面更新时自动检测新内容
4. 命中高风险线索后自动固证
5. 通过邮箱或钉钉机器人推送通知
6. 用户进入工作台查看案件详情
7. 后续文书与审核动作与主动取证链路一致

## 6. 明确的产品边界

这是项目最重要的约束之一。

### 6.1 不做的事

当前版本不做：

- 不直接输出“已构成侵权”的最终法律结论
- 不自动替代律师向外发送正式法律文书
- 不自动代替用户向主管部门发起正式举报
- 不做无边界的开放式聊天助手
- 不做全网无限制抓取
- 不承诺司法机关一定采信平台生成的证据

### 6.2 应使用的表述

系统对外输出建议统一使用：

- `疑似侵权内容`
- `高风险侵权线索`
- `相似度较高`
- `可能落入保护范围`
- `建议人工复核`

避免使用：

- `已侵权`
- `构成侵权`
- `法院可直接采信`
- `可直接自动起诉`

## 7. 平台形态共识

当前确定采用：`浏览器插件 + 工作台 + 后台 Agent`

### 7.1 浏览器插件

负责：

- 一键取证
- 当前页面信息采集
- 用户快速备注
- 将证据提交到工作台

### 7.2 工作台

负责：

- 案件管理
- 权利资产管理
- 证据包查看
- 风险分析结果展示
- 文书模板选择与生成
- 人工审核流
- 导出与归档

### 7.3 后台 Agent

负责：

- 自动巡检任务
- 相似度比对任务
- 风险评级任务
- 通知触发
- 文书生成编排
- 模板优化与案例学习

## 8. Hermes 在系统中的角色

Hermes 适合作为：

- 后台编排引擎
- 巡检与通知任务调度器
- 文书生成任务执行体
- 受控学习与模板优化的执行层

Hermes 不适合作为：

- 面向终端用户的开放式自由聊天入口
- 唯一权限控制系统
- 直接暴露所有底层能力给普通用户的总入口

因此最终架构共识是：

`用户/机器人/插件 -> 平台权限网关 -> 工作台 API -> Hermes Worker`

## 9. 关于“学习”机制的共识

当前方向允许做“学习”，但不能做成失控的“自我进化”叙事。

更推荐的表达是：

- `基于历史案件与人工反馈的受控学习`
- `案例驱动的持续优化机制`
- `模板与规则的可回滚迭代`

### 9.1 可以学习的内容

- 哪些证据字段更关键
- 哪类案件适合优先生成哪种模板
- 哪些表述容易过强，需要降级
- 哪些线索更值得优先推送
- 哪些案件摘要结构更利于法务审核

### 9.2 不应交给系统自动决定的内容

- 是否构成侵权的最终认定
- 是否自动发正式法律文书
- 是否自动代表用户举报主管部门
- 是否自动覆盖所有旧模板
- 是否自动越过人工审核

## 10. 通知机制共识

第一阶段以两种通知渠道为主：

- `邮箱`
- `钉钉机器人`

后续可扩展：

- 飞书
- 企业微信
- 短信
- Slack / Discord 等

### 10.1 邮箱策略

- 平台统一配置一个发件邮箱
- 用户只填写收件邮箱
- 邮件内容以风险摘要和工作台链接为主
- 大型证据包不直接作为主附件发送，建议在工作台下载

### 10.2 钉钉策略

- 优先做受限动作型机器人
- 适合推送案件摘要与风险通知
- 可通过卡片按钮触发“查看详情”“生成律师函”“生成举报材料”等受控动作

## 11. 权限与安全边界共识

### 11.1 设计原则

- 用户和管理员必须隔离
- 案件和证据必须按品牌/组织/角色隔离
- 所有高风险动作必须留痕
- 文书输出必须经过人工审核链路

### 11.2 推荐角色

- `viewer`：查看摘要与部分证据
- `operator`：发起取证、触发生成、提交审核
- `reviewer`：审核文书、修正内容、确认导出
- `admin`：配置站点、模板、通知和用户

### 11.3 安全实现原则

- 用户不能直接自由调用 Hermes
- 机器人只开放受限命令
- 权限控制由平台外层系统承担
- Hermes 只承担执行编排与任务处理
- 全部生成和修改记录可审计、可回滚

## 12. MVP 范围

当前最适合比赛的 MVP 是：

1. 浏览器插件一键取证
2. 证据包生成与案件归档
3. 基于规则或模型的风险分析与相似度评分
4. 邮箱/钉钉的风险通知
5. 律师函/投诉函/举报材料/证据目录的初稿生成
6. 人工审核后导出

## 13. 后续增强能力

后续可以扩展但不应作为第一版主战场的功能：

- 复杂多站点自动盯站
- 深度行业/国别模型
- 司法存证、公证接口对接
- 企业微信/飞书/短信等多渠道通知
- 案件协作与法务审批流
- 历史案件训练的模板优化引擎

## 14. 当前对外最稳的定位表述

推荐统一使用以下定位：

> 证证鸽是一套面向知识产权侵权响应场景的智能取证与处置辅助平台，通过主动取证、自动监测、证据固化、风险分析、消息推送与文书初稿生成，帮助品牌方、创作者和法务团队更高效地完成侵权线索处置闭环。
