## 1. 当前结论

证证鸽已经从“纯文档阶段”进入“可运行骨架 + 已打通核心联调链路阶段”。

当前已经具备：

- 独立 Git 仓库
- 主站 Landing Page
- 工作台基础壳子
- 浏览器插件基础取证能力
- FastAPI 后端基础接口
- SQLite 持久化
- 插件 -> 后端 -> 工作台 的最小链路
- 草稿、监控、接收方式的真实 API 与页面
- Production 下可运行的主站与工作台

但仍未达到完整产品化状态，主要还缺：

- 自动监测任务真实执行
- LLM 驱动的摘要与文书生成

## 3. 已验证链路

已实测通过的链路：

1. 后端健康检查通过
2. Web lint / build 通过
3. 插件 build 通过
4. API pytest / smoke test 通过
5. 手工向 `/api/v1/evidence/intake` 提交插件同形 payload 可成功创建案件和证据包
6. 工作台 `/workspace/cases` 已切换到真实 API 数据源
7. 工作台 `/workspace/cases/[caseId]` 可显示真实案件详情与证据包
8. 工作台 `/workspace/drafts` 可生成草稿并跳转详情页
9. 工作台 `/workspace/drafts/[draftId]` 可导出草稿
10. 工作台 `/workspace/monitoring` 可执行监控任务
11. 工作台 `/workspace/notifications` 可发送测试消息
12. Landing Page、文档入口、工作台入口已通过浏览器验收
13. `next start` 生产态链路已验证可用

## 4. 当前缺口

### P0 缺口

- 自动监测仍未接入真实抓取与建案
- 文书内容仍是模板拼装版，未接 LLM
- 证据包文件仍未在前端提供预览与下载

### P1 缺口

- 自动监测任务真实执行
- 邮件真实 SMTP 联调
- 即时提醒渠道联调
- 文书模板真实渲染与 docx/pdf 导出
- 审核任务与通知日志闭环

### P2 缺口

- Hermes 完整编排接入
- 历史案例驱动优化
- 飞书 / 企业微信扩展
- 海外站点扩展

## 5. 当前风险

1. 前后端字段契约仍在收敛期，后续需要继续约束返回结构。
2. 当前 API 虽已接 SQLite 与多模块接口，但仍偏 demo 形态，尚未进入正式业务模型。
3. 插件当前是 popup 模式，尚未进入更强的 content script / background 体系。

## 6. 下一步优先级

1. 自动监测任务真实执行
2. LLM 文书初稿与案件摘要生成
3. 证据包预览与下载
4. 登录 / 权限 / 审计
5. 接口 / 数据库 /任务拆分继续跟文档基线对齐
