# 证证鸽实施状态

> 更新时间：`2026-04-11`  
> 用途：记录当前开发进度、已交付能力、已验证链路与剩余缺口，作为产品推进和联调收口依据。

## 1. 当前结论

证证鸽已经从“纯文档阶段”进入“可运行骨架 + 初步联调阶段”。

当前已经具备：

- 独立 Git 仓库
- 主站 Landing Page
- 工作台基础壳子
- 浏览器插件基础取证能力
- FastAPI 后端基础接口
- 插件 -> 后端 -> 工作台 的最小链路

但仍未达到最终可交付状态，主要还缺：

- 持久化存储
- 更完整的案件详情与证据展示
- 更强的插件结果反馈
- 更多真实数据流而非占位逻辑

## 3. 已验证链路

已实测通过的链路：

1. 后端健康检查通过
2. Web lint / build 通过
3. 插件 build 通过
4. API pytest / smoke test 通过
5. 手工向 `/api/v1/evidence/intake` 提交插件同形 payload 可成功创建案件和证据包
6. 工作台 `/workspace/cases` 已切换到真实 API 数据源

## 4. 当前缺口

### P0 缺口

- 案件与证据仍需持久化，避免重启丢数据
- 案件详情页仍有部分字段是占位或弱映射
- 插件提交成功后仍缺少更明确的案件跳转入口

### P1 缺口

- 自动监测任务真实执行
- 钉钉 / 邮件真实通知联调
- 文书模板真实渲染与导出
- 审核流与状态机完整闭环

### P2 缺口

- Hermes 完整编排接入
- 历史案例驱动优化
- 飞书 / 企业微信扩展
- 海外站点扩展

## 5. 当前风险

1. 前后端字段契约仍在收敛期，后续需要继续约束返回结构。
2. 当前 API 仍偏 demo 形态，未接数据库前不适合长期演示。
3. 插件当前是 popup 模式，尚未进入更强的 content script / background 体系。

## 6. 下一步优先级

1. API 持久化
2. 案件详情页真实证据包展示
3. 插件成功页与案件跳转
4. 通知通道联调
