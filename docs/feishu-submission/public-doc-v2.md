# 证证鸽｜AI 知识产权侵权响应平台

## 让疑似侵权页面，更快变成可行动的案件材料

<image token="WhQMbTCmpoFO1qxx7FFc8JZSnVg" width="1178" height="440" align="center"/>

证证鸽是一套面向**品牌方、法务团队与知识产权服务场景**的 AI 知识产权侵权响应平台。  
它的核心目标不是“多做几个页面”，而是把分散、易失的网页线索，快速转化为**标准化证据包、结构化案件材料、风险摘要与可审核的法律文书草稿**。

> 当前版本重点强化的是一条真正可执行的链路：**发现线索 -> 页面取证 -> 证据沉淀 -> 自动生成草稿 -> 人工审核导出**。

## 项目聚焦的问题

- 发现疑似侵权页面后，往往来不及完整留证
- 证据分散在截图、链接和聊天记录里，难以进入后续处置流程
- 监测、取证、研判、文书准备之间缺少连续闭环

## 当前版本已实现的核心能力

| 模块 | 当前实现 |
| --- | --- |
| 浏览器插件取证 | 一键提交 URL、标题、正文、HTML、截图和备注 |
| 案件工作台 | 案件列表、案件详情、风险评分、处理建议 |
| 证据包沉淀 | 证据包详情、截图预览、HTML 片段、基础证据目录 |
| 自动监控 | 可配置目标地址、关键词、频率和风险阈值 |
| 取证后一键生成 | 取证完成后自动生成律师函 / 平台投诉函等法律文书草稿 |
| Word 导出 | 将草稿整理为可继续审核的 `.docx` 文档 |
| 后端编排 | 通过 Hermes 编排抓取、摘要、文书与通知任务 |

## 业务闭环

从发现线索开始，到证据沉淀、风险研判、材料生成，证证鸽希望把“网页侵权线索”变成“可继续推进、可继续审核、可继续导出的案件材料”。

## 功能展示

### 1. 一键取证，把网页内容转成标准证据包

浏览器插件发起取证后，系统会统一采集和沉淀：

- 当前 URL
- 页面标题
- 页面正文
- 原始 HTML
- 页面截图
- 抓取时间
- 请求编号与备注

系统收到取证请求后，会自动创建案件并生成对应证据包；当证据达到基础完整度后，会继续触发默认法律文书草稿生成，为后续研判和处置提供统一输入。

<image token="I8HWbC0xaoD7FNxX9uscTIcRnZc" width="1688" height="1004" align="center"/>

### 2. 案件工作台，把线索变成可处理事项

工作台中的案件详情页已经支持：

- 风险评分
- 证据包数量
- 目标对象
- 处理建议
- 基础证据目录

这一步的意义，不只是“看见一个可疑页面”，而是把它升级成团队可以继续推进的案件条目。

<image token="ElfXbF7P5oUouixSQRucJ1gCn8D" width="1673" height="1577" align="center"/>

### 3. 自动监控巡检，把一次性处理升级成持续观察

除主动取证外，当前版本也已经支持监控任务：

- 配置目标页面或站点
- 设置品牌关键词
- 设置巡检频率
- 设置风险阈值
- 命中后进入后续取证与处理流程

这让系统具备了持续发现高风险线索的能力，而不只是事后补救。

<image token="HFt4bOnauo0x5Jx4VTEcYOilndC" width="1688" height="1004" align="center"/>

### 4. 文书草稿生成，把证据继续推进到处置动作

案件与证据沉淀后，系统会优先生成一份默认法律文书草稿，工作台也支持继续切换或补充生成，例如：

- 律师函初稿
- 平台投诉函
- 举报材料草稿

当前版本的目标，不是替代律师做最终定稿，而是把“先取证、再整理、再起草”的前置工作尽量自动化：

- 参照现有法律文书模板的语言风格输出结构化草稿
- 把页面取证信息、证据要点和案件上下文自动填入
- 导出为可继续审核和修改的 Word 文档

当前定位仍然是**提供可审核、可继续完善的法律文书草稿**，帮助法务团队更快完成前期整理，而不是直接替代人工输出最终法律结论。

<image token="DxB8b0qBboEmttxpMqXcQIR6nnd" width="2595" height="1495" align="center"/>

### 5. 证据补强与处置提效

在真实的知识产权处置场景里，网页截图通常只是起点，后续往往还需要更多补强材料来形成完整证据链，例如：

- 可信时间戳
- 商标注册证
- 录频材料
- 线上购买侵权商品的订单与物流记录
- 涉案净额相关材料

当前版本已经完成了**网页证据的标准化沉淀**，包括截图、HTML、正文、抓取时间和案件结构化整理。  
但在取证渠道上，需要明确说明：

> **由于技术及国家认证要求等因素，暂时还未接入可信时间戳的 API 渠道。**

因此，当前证据更适合用于内部研判、案件归档、草稿生成和前期沟通准备；如需进一步进入正式司法或高强度对外维权流程，仍建议在后续补入更强的固证材料。

其中，`可信时间戳` 的价值尤其明确：

- 用于补强网页内容形成与留存时间
- 降低人工整理取证材料的成本
- 提升后续投诉、发函和内部流转的效率

正式投产后，有望在**「取证固证」环节**加入可信时间戳功能，以实现更精准的取证固证。当前项目和程序逻辑，也会持续朝这个方向预留和靠拢。

## 技术实现简述

### 技术栈

| 层级 | 当前采用 |
| --- | --- |
| 浏览器插件 | Plasmo |
| Web 工作台 | Next.js |
| 后端 API | FastAPI |
| 数据存储 | SQLite |
| 页面抓取 | PlaywrightWorker |
| 生成能力 | OpenAI-compatible LLM（MiMo） |
| 通知能力 | NotificationAdapter（邮件 / Webhook） |
| 工作流编排 | HermesOrchestrator |

### 后端架构

当前后端并不是单接口集合，而是围绕案件、证据、监控、文书和通知构建的一套可编排服务结构。

<whiteboard token="FBBJwBzLKhIl7ZbMpe6cJtfxnec" align="left"/>

### Hermes 在当前版本中的作用

Hermes 主要承担三类编排职责：

1. 取证链路编排：插件提交后触发取证工作流
2. 监控链路编排：命中阈值后生成案件摘要并触发通知
3. 文书链路编排：生成律师函、投诉函等可审核草稿

也就是说，Hermes 在这里不是单独对话入口，而是后端任务编排中枢，用来把抓取、研判、通知和文书生成串成连续流程。

<whiteboard token="Pl0ewaGkJhAkZCbMOuQcF12lnJe" align="left"/>

## 当前阶段的交付结果

当前版本已经具备以下真实能力：

- 有真实可运行的前端页面和工作台
- 有真实后端接口和运行时模块
- 有真实案件、证据包、监控任务和文书草稿数据
- 有真实的编排层，能够串联取证、摘要、通知和文书能力
- 有“取证后自动生成默认草稿”的基础链路

## 总结

证证鸽当前聚焦的，不是抽象的“AI + 知识产权”，而是一个具体且高频的动作：

> 当团队发现一个疑似侵权页面时，如何在最短时间里把它变成可留存、可管理、可继续推进，并能直接进入法律文书起草的证据和案件材料。

围绕这个目标，当前版本已经完成了从**取证入口、案件沉淀、自动监控到草稿生成与导出**的基础闭环，并通过 Hermes 编排能力把多步任务真正串联起来。
