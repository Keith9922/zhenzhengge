# 证证鸽项目资料包

> 本文档用于集中查阅证证鸽项目当前阶段的核心方案。内容整合自产品共识、品牌方案、用户流程与技术路线三份文档，适合用于内部讨论、方案对齐、海报准备与答辩梳理。

## 1. 项目概览

### 1.1 项目名称

`证证鸽`

### 1.2 产品副标题

`证证鸽：侵权线索固证与文书辅助平台`

### 1.3 一句话介绍

> 证证鸽通过一键取证、自动固证、风险分析、消息推送与文书辅助，帮助品牌方和法务团队更高效地完成侵权线索处置闭环。

### 1.4 Slogan

`先留证，再行动。`

## 2. 项目为什么成立

知识产权侵权处理中，最容易被忽视、但最关键的前置环节，不是最后起诉，而是最开始的：

- 发现线索后能不能及时留证
- 留下来的证据是否标准化
- 证据能不能快速进入后续法务流程
- 长期监测是否能持续、稳定、低成本

证证鸽的核心判断是：

> 先把“侵权线索”转成“标准化证据包”和“结构化案件材料”，再去谈后续投诉、发函、举报和维权。

因此它不是纯监测工具，也不是纯文书生成器，而是一条围绕侵权响应场景展开的工作流平台。

## 3. 核心定位

证证鸽是一套面向知识产权侵权响应场景的智能取证与处置辅助平台。

它的核心价值是把分散、易失的线上侵权内容，快速转化为：

- 可留存的证据包
- 可管理的案件材料
- 可推送的风险事件
- 可进入人工审核流程的文书初稿

## 4. 产品重心与边界

### 4.1 当前重心

当前版本明确以`取证固证`为重心。

后续法律文书、投诉材料、举报材料等能力，属于围绕取证后的自然延伸，用来形成完整闭环。

### 4.2 不做的事

当前版本不直接做：

- 不直接输出“已构成侵权”的最终法律结论
- 不自动替代律师向外发送正式法律文书
- 不自动代表用户向主管部门正式举报
- 不做无限制的开放式法律聊天助手
- 不承诺生成结果可被法院或主管机关直接采信

### 4.3 系统应使用的表述

系统输出统一建议使用：

- `疑似侵权内容`
- `高风险侵权线索`
- `相似度较高`
- `可能落入保护范围`
- `建议人工复核`

## 5. 两条核心产品链路

### 5.1 主动取证模式

适用于用户自己已经发现疑似侵权页面的场景。

流程如下：

1. 用户在网页中发现疑似侵权内容
2. 点击浏览器插件，一键发起取证
3. 系统采集页面信息并生成标准化证据包
4. 系统执行相似度比对与风险评级
5. 生成案件摘要、证据目录与建议动作
6. 用户可选择生成律师函、投诉函、举报材料等文书初稿
7. 法务或审核人进行修订、确认与导出

### 5.2 自动监测模式

适用于用户希望平台持续盯站、自动发现新风险的场景。

流程如下：

1. 用户在工作台配置监控任务
2. 平台对指定网站、页面或店铺定时巡检
3. 一旦页面出现更新，系统自动抓取新内容
4. 命中高风险线索后自动执行固证
5. 系统通过邮箱或钉钉机器人推送通知
6. 用户进入工作台查看案件详情
7. 后续文书与审核流程与主动取证模式一致

## 6. 用户能得到什么

### 6.1 证据层

系统可沉淀出标准化证据包，内容包括：

- 当前 URL
- 页面标题
- 全页截图
- 当前视口截图
- 原始 HTML
- 页面文本提取结果
- 图片链接
- 抓取时间
- 哈希值
- 操作日志

### 6.2 分析层

系统可输出：

- 相似度结果
- 风险评级
- 疑似侵权点说明
- 建议动作
- 简单涉案金额估算

### 6.3 处置层

系统可辅助生成：

- 律师函初稿
- 平台投诉函
- 举报材料草稿
- 侵权分析说明
- 证据目录

## 7. 品牌方案

### 7.1 为什么叫“证证鸽”

- 与“证据、取证、固证、举证”天然关联
- 具备记忆点，适合比赛传播
- 比“公平鸽”更贴近当前产品核心
- 既可做插件 IP，也可做机器人形象

### 7.2 IP 形象建议

证证鸽不是法官，也不是律师，而是一只：

- 负责巡查的鸽子
- 负责送证的鸽子
- 负责提醒风险的鸽子
- 负责把案件材料送入后续流程的助手

建议视觉风格：

- 蓝白配色为主
- 带有放大镜、证据包、时间戳、信封、盾牌等元素
- 可爱但克制，亲和但不轻浮

### 7.3 推荐海报标题

主标题：

`证证鸽`

副标题：

`知识产权侵权取证、固证与处置辅助平台`

展示标题备选：

- `证证鸽：让侵权页面变成可行动的证据`
- `证证鸽：从侵权线索到法务处置闭环`
- `证证鸽：一键取证，自动固证，辅助处置`

## 8. 模块命名

为了统一产品表达，建议采用“鸽 + 功能”的命名方式：

- `鸽眼`：监测与识别
- `鸽证`：取证与固证
- `鸽信`：消息推送
- `鸽函`：文书生成
- `鸽审`：人工审核
- `鸽库`：案件与证据归档

## 9. 平台形态

当前建议采用：

`浏览器插件 + Web 工作台 + 后台 Hermes Agent`

### 9.1 浏览器插件

负责：

- 一键取证
- 页面信息采集
- 用户补充备注
- 将证据提交到工作台

### 9.2 Web 工作台

负责：

- 权利资产管理
- 案件管理
- 证据包查看
- 监控任务配置
- 模板选择与文书生成
- 审核与导出

### 9.3 后台 Hermes Agent

负责：

- 定时巡检任务
- 页面更新后的任务编排
- 相似度分析与风险摘要
- 文书生成调度
- 消息推送触发
- 模板优化与案例学习

## 10. 为什么要用 Hermes

Hermes 适合作为平台内部的：

- 执行编排引擎
- 巡检任务调度器
- 通知触发器
- 文书生成 Worker

但 Hermes 不应直接作为：

- 面向终端用户的开放式自由聊天机器人
- 唯一权限控制系统
- 直接暴露全部底层能力的入口

因此推荐结构是：

`用户/插件/机器人 -> 平台权限网关 -> 工作台 API -> Hermes Worker`

## 11. 通知方案

当前第一阶段重点做两种通知渠道：

- `邮箱`
- `钉钉机器人`

### 11.1 邮箱方案

- 平台统一配置一个发件邮箱
- 用户只需填写收件邮箱
- 邮件以风险摘要 + 工作台链接为主
- 大证据包建议在工作台中查看或下载

### 11.2 钉钉方案

钉钉机器人建议做成`受限动作型机器人`，主要支持：

- 接收案件预警
- 查看案件摘要
- 跳转工作台
- 触发“生成律师函”
- 触发“生成举报材料”
- 标记已收到 / 待处理

不建议第一版做开放式自由聊天。

## 12. 权限与安全边界

### 12.1 基本原则

- 用户与管理员必须隔离
- 案件与证据必须按组织/品牌/角色隔离
- 所有生成和修改动作必须留痕
- 高风险动作必须人工确认

### 12.2 推荐角色

- `viewer`：查看案件摘要与部分证据
- `operator`：发起取证、触发生成、提交审核
- `reviewer`：审核文书、修正内容、确认导出
- `admin`：配置模板、站点、通知策略和用户

### 12.3 权限策略

- 用户不能自由调用底层 Agent
- 机器人只开放受限命令
- 权限控制由平台外层系统承担
- Hermes 只负责执行，不负责最终权限裁定

## 13. 文书策略

文书属于后续连带能力，不是产品唯一核心，但必须形成闭环。

当前推荐优先做：

- 律师函
- 平台投诉函
- 举报材料草稿
- 侵权分析说明
- 证据目录

推荐生成逻辑：

`案件结构化数据 -> 模板填充 -> 大模型润色 -> 人工审核`

这样会比“直接对着网页内容生成文书”更稳定。

> 证证鸽不是一个自动判定侵权的机器人，而是一套面向知识产权侵权响应场景的智能取证与处置辅助平台。它通过浏览器插件解决网页证据易失问题，通过工作台完成案件管理、风险分析和文书生成，通过后台 Hermes Agent 完成自动巡检、通知和任务编排，最终把分散的侵权线索沉淀为可管理、可审核、可行动的案件材料。

## 19. 参赛评估与调整建议

### 19.1 建议申报的主赛道

如果参赛需要在三条方向中明确选择一条，当前最推荐的主赛道是：

`AI + 知识产权侵权智能监测`

原因是证证鸽当前的核心价值链条仍然是：

- 发现侵权线索
- 抓取页面证据
- 固化证据
- 进行风险分析
- 触发后续动作

这与专家分享中强调的“数据采集 -> 发现定位 -> 分析比对 -> 风险预判 -> 自动取证/固证 -> 后续管理”的主线高度一致。

### 19.2 可以在答辩中强调的延展赛道

虽然主赛道建议选“侵权智能监测”，但在答辩时应主动说明项目同时覆盖另外两条方向：

- `AI + 知识产权法律服务`
  - 体现在律师函、投诉函、举报材料等文书初稿能力
  - 体现在人工审核与法务协同链路

- `海外知识产权风险预警`
  - 体现在自动巡检、风险通知、品牌词/行业词监控
  - 如果后续聚焦跨境电商、TRO 等场景，这一条会更强

### 19.3 与主讲人要求的匹配度

当前方案与主讲人分享内容的匹配度较高，原因主要有以下几点：

1. `符合“先监测、再取证、再处置”的整体方向`
2. `抓住了专家反复强调的“自动取证与证据固化”这一高价值节点`
3. `没有把重点放在最难的发明专利侵权自动判定上，路径更现实`
4. `文书生成被放在取证之后，符合专家关于“通用模板先做起来”的建议`
5. `支持持续监控、预警推送，贴合专家提到的 7x24 小时监测思路`

### 19.4 结合比赛介绍的优势判断

根据本地 Playwright 材料中提取到的比赛介绍文字，本次比赛强调的关键词包括：

- `AI × 知识产权`
- `法律思维`
- `创造与守护`
- `开放式路演与互动展示`
- `技术、创意、潜力`
- `后续落地支持`

证证鸽与这些关键词的契合度较高：

- 它有明确的“守护创新成果”叙事
- 它不是空泛法律问答，而是可操作的工作流系统
- 浏览器插件 + 工作台 + 机器人通知很适合线下展示
- “证证鸽”这一 IP 也有利于增强记忆点与创意表达
- 从商业化看，有品牌方、法务、跨境卖家等明确付费对象

### 19.5 拿奖潜力判断

如果按当前方向继续收敛并做出完整 Demo，这个项目是有拿奖潜力的。

更现实的奖项倾向判断如下：

- `最佳潜力奖`：潜力最大
  - 原因是商业化路径清晰，应用场景真实，后续能延展为企业服务产品

- `最佳创意奖`：也有机会
  - 原因是“证证鸽”IP、插件 + 工作台 + 机器人通知的形态较有展示性

- `最佳技术奖`：有机会，但要求更高
  - 取决于你们是否能把自动巡检、风险评分、证据包生成、消息触发这些链路真正跑通，而不只是做页面壳

### 19.6 当前方案最需要继续调整的地方

为了更接近获奖状态，建议继续做以下收敛：

1. `聚焦单一高价值场景`
   - 当前最推荐的第一场景：`淘宝 / 拼多多 / 京东商品页 + 品牌官网页面` 的商标近似、冒用、变体侵权监测
   - 典型问题包括：品牌名替换、近似商标、山寨命名、图文混用、店铺页侵权展示
   - 不要第一版同时覆盖所有知识产权类型和所有网站

2. `把“取证固证”做成最强演示点`
   - 插件一键抓取
   - 证据包标准化
   - 时间留痕
   - 哈希校验
   - 工作台一键查看

3. `把“自动监测”做成可信增强能力`
   - 可以先支持少量指定站点和定时巡检
   - 不要一开始讲成全网万能监测

4. `把文书定义为“法务可用初稿”，而不是最终正式法律结论`
   - 这会让评委更容易接受

5. `做出一条完整可演示链路`
   - 打开淘宝 / 拼多多 / 京东或品牌官网中的疑似侵权页面
   - 插件一键取证
   - 工作台生成证据包和风险评分
   - 钉钉 / 邮件收到预警
   - 生成律师函或平台投诉函初稿

### 19.7 推荐第一版演示场景

建议第一版路演时，统一聚焦下面这个场景：

> 某品牌方已经在系统中录入自己的商标或品牌词，运营人员在电商平台或品牌官网发现疑似冒用、近似命名或仿冒页面后，使用证证鸽插件一键取证；系统生成证据包、完成风险分析，并通过工作台生成律师函或平台投诉材料初稿，最后进入人工审核。

这个场景的优势是：

- 贴近淘宝、拼多多、京东、品牌官网等真实环境
- 不依赖复杂跨境法域和 TRO 背景知识
- 更容易讲清“商标冒用/近似侵权”这一类大众可理解的问题
- 更适合现场 Demo 演示

### 19.8 关于“全网监测”的对外讲法

推荐统一表述为：

> 证证鸽支持对用户指定的网站、页面、店铺或栏目进行持续巡检与更新监测。

不建议第一版直接说：

- 全网自动监测所有侵权内容
- 无限制监控整个互联网

因为更稳的表达应强调：

- 名义上具备面向全网扩展的能力
- 当前版本通过“指定站点 / 指定页面 / 指定店铺 / 指定品牌词”的方式提升可行性与准确性

### 19.9 举报入口设计建议

举报功能建议保留为`后续可选动作入口`，而不是自动执行动作。

推荐在工作台和机器人中都预留：

- `生成举报材料`
- `查看举报材料草稿`
- `提交人工审核`
- `导出后自行上报`

这样既保留闭环，也不会引入“系统自动代替用户向主管部门发正式举报”的风险。

### 19.10 一句话结论

证证鸽当前方案是符合主讲人方向要求的，也符合比赛“AI + 知识产权 + 可展示 + 可落地”的基调。  
如果你们能把场景收敛、Demo 做顺、边界讲稳，它是一个有现实拿奖希望的题目。
