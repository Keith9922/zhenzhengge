# 证证鸽与 Page Agent 适配评估

> 文档类型：技术取舍记录  
> 版本：v0.1  
> 更新时间：2026-04-11  
> 结论：`P0 不接入，保留 P1/P2 扩展入口`

## 1. 评估背景

当前项目核心目标是先跑通这条链路：

`插件取证 -> 案件创建 -> 证据包归档 -> 草稿生成 -> 监控任务 -> 接收方式测试`

用户额外提出希望评估下列开源项目是否可纳入：

- `https://github.com/alibaba/page-agent`

## 2. Page Agent 适合解决什么问题

Page Agent 更偏向：

- 浏览器里的自然语言操作
- 表单自动填写
- 页面动作执行
- 用户用一句话驱动浏览器完成点击和输入

这类能力更接近：

- 智能浏览器助手
- 自动化操作代理
- 页面级交互执行层

## 3. 当前项目的主问题是什么

证证鸽当前 P0 的主问题不是“如何自动填表”，而是：

- 如何把网页线索快速转成标准化证据包
- 如何把案件和草稿链路跑通
- 如何把监控与接收方式接成真实 API

所以 P0 的主战场是：

- 插件采集
- 后端持久化
- 工作台页面
- 草稿与审核
- 监控任务

## 4. 是否建议在 P0 接入

不建议。

## 5. 原因

### 5.1 不是当前主链路阻塞点

当前最关键的阻塞点是业务闭环，不是页面内自动操作。

### 5.2 会抬高调试复杂度

如果现在接入 Page Agent，会把调试范围从：

- 插件
- 工作台
- 后端 API

扩成：

- 浏览器动作代理
- 指令理解
- 页面执行稳定性

这会稀释 P0 进度。

### 5.3 价值在 P1/P2 更明显

Page Agent 更适合作为：

- 自动化补录
- 自动化表单填写
- 指令式页面操作
- 后续“让 Bot 帮用户执行页面动作”的增强模块

而不是当前版本的核心依赖。

## 6. 建议结论

### P0

不接入 Page Agent。

### P1/P2 可考虑的接入点

1. 在工作台中增加“自然语言填写表单”
2. 在插件中增加“自动抓取当前页指定字段”
3. 在后续举报材料或平台投诉页面中做半自动填报

## 7. 当前建议

P0 继续坚持现有技术路径：

- 插件：`Plasmo`
- 工作台：`Next.js`
- 后端：`FastAPI`
- 编排：`Hermes Agent`
- 抓取：`Playwright`

Page Agent 暂时只作为后续增强项记录，不纳入当前主分支开发范围。
