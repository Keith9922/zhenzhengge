# 证证鸽路演PPT构建文档（V1）

> 文档用途：定义“证证鸽”路演PPT的统一风格与页面规范，确保制作过程可复用、可验收。  
> 适用对象：产品、设计、研发、路演讲解人。  
> 版本时间：2026-04-21。

---

## 1. 风格规范

### 1.1 视觉定位

整套PPT采用“专业技术产品路演风格”，核心目标是：

- 看起来是**真实可落地的工程产品**，而不是概念海报
- 兼顾法务场景的严肃感与AI产品的效率感
- 让评委能快速抓到“问题-方案-落地-边界-路线图”

### 1.2 色彩规范

主色建议：

- 主背景：浅冷灰蓝 `#F2F6FC`
- 主文本：深海军蓝 `#11284C`
- 强调蓝：`#1E6BFF`
- 协同绿：`#1FBF9A`
- 风险提示橙：`#F39C5A`

使用原则：

- 每页最多 1 个主强调色 + 1 个辅助强调色
- 大段正文避免纯黑，优先深蓝灰，提高投影可读性
- 卡片底色统一浅色，边框统一细线，不做重阴影

### 1.3 字体规范

- 中文主字体：苹方 / 思源黑体（设备缺失时可回退）
- 英文与代码字段：Aptos / 等宽字体仅用于端点、路径等

字号建议：

- 页面主标题：48-56
- 二级标题：30-36
- 卡片标题：22-28
- 正文：15-20
- 备注/来源：11-13

### 1.4 版式规范

- 统一 16:9
- 所有页面保留顶部导航线（章节名 + 页码）
- 左上角统一章节标签（如“结构图 01”）
- 页面下边统一弱化品牌角标，不抢内容
- 一页信息不超过 1 个核心结论 + 3 个支撑点

### 1.5 图文策略

- **先图后字**：流程与架构优先画图，不堆长段文字
- **真实优先**：前后端页面用真实截图，不用示意图代替
- **可验证优先**：所有能力表述对应当前仓库文档中的已实现项

---

## 2. 页面结构规范（建议 12 页）

> 叙事顺序：
> 封面 -> 问题 -> 产品主张 -> 业务闭环 -> 工程结构 -> 真实界面 -> 协调机制 -> 落地价值 -> 风险边界 -> 路线图

### P1 封面页

页面目标：30秒内讲清项目身份和一句话价值。

必须放：

- 项目名：证证鸽
- 副标题：AI知识产权侵权响应平台
- Slogan：先留证，再行动
- 一句话价值：把疑似侵权线索变成可行动案件材料

建议讲稿：

- “我们做的不是单点检测，而是侵权响应的工作流平台。”

---

### P2 问题定义页

页面目标：解释“为什么必须做这个产品”。

必须放：

- 证据易失（页面会变）
- 处置链路割裂（监测、取证、文书分散）
- 法务前置整理成本高（效率瓶颈）

建议文案：

- “侵权处理最大的损耗，发生在最开始的证据整理，而不是最后起诉。”

---

### P3 产品主张页

页面目标：定义产品边界和价值主线。

必须放：

- 不是“侵权判定机器人”
- 是“取证 + 案件结构化 + 文书草稿”的工作流平台
- 坚持“AI初稿 + 人工审核”

建议文案：

- “先把线索变成标准化证据包，再谈后续投诉与维权动作。”

---

### P4 双链路闭环页

页面目标：展示业务流程的完整性。

必须放：

- 主动取证链路：插件触发 -> 证据包 -> 案件 -> 草稿 -> 审核导出
- 自动监控链路：任务配置 -> 巡检命中 -> 自动固证 -> 通知 -> 处置
- 两条链路共用一个案件出口

建议图形：

- 双泳道流程图（左：主动；右：自动；底部汇聚到案件工作台）

---

### P5 当前交付成果页

页面目标：证明“不是概念产品，已有可运行结果”。

必须放（使用已验证口径）：

- 已实测通过关键链路（至少 11 条）
- 前后端+插件均可运行
- 三角色鉴权已落地（viewer/operator/admin）

建议文案：

- “当前阶段已进入可运行骨架 + 核心联调跑通状态。”

---

### P6 结构图 01：前后端总体架构图

页面目标：回答“系统结构长什么样”。

必须放：

- 用户入口（插件 / 工作台）
- 前端层（Next.js + Plasmo）
- 后端网关（FastAPI）
- 编排与服务层（Hermes / Playwright / LLM / 通知）
- 数据层（SQLite + 文件存储）
- 数据流向箭头（请求、编排、入库）

建议文案：

- “前端负责触发与展示，后端负责编排与治理。”

---

### P7 结构图 02：前后端长什么样（真实界面）

页面目标：回答“不是Demo壳，实际界面已经存在”。

必须放：

- 前端主站截图
- 前端工作台截图（案件页或证据页）
- 后端 API Docs 截图（FastAPI /docs）
- 每张图下方标签说明

建议文案：

- “这三张都来自当前仓库实际运行截图，支持现场复现。”

---

### P8 结构图 03：后端协调架构（Hermes 编排）

页面目标：解释后端多模块如何协同。

必须放：

- Intake 接入
- Evidence Service
- Hermes Orchestrator（中心）
- Playwright 抓取
- 风险评分
- 通知 + 草稿生成
- 最终写入案件/证据/日志

建议文案：

- “Hermes 的价值是把离散动作变成可追踪的连续任务。”

---

### P9 能力矩阵页

页面目标：把能力按层整理，降低理解成本。

必须放：

- 取证层：URL/正文/HTML/截图采集
- 研判层：风险评分、监控任务、通知日志
- 处置层：草稿生成、富文本编辑、docx导出

建议文案：

- “从取证到文书，关键模块已具备。”

---

### P10 目标用户与场景页

页面目标：说明商业与应用落地可能性。

必须放：

- 品牌法务团队
- 跨境电商运营
- 知识产权服务机构

建议文案：

- “优先服务高频、刚需、可验证场景。”

---

### P11 风险边界页

页面目标：主动降风险，增强可信度。

必须放：

- 法律边界：不自动宣称侵权成立
- 产品边界：不自动代表用户完成正式投诉
- 工程边界：用户体系/细粒度ACL仍在补齐

建议文案：

- “我们强调可验证事实，不做夸大承诺。”

---

### P12 路线图页

页面目标：交代下一阶段推进计划。

必须放：

- P0：时间戳策略、模板补齐、可观测性
- P1：审核任务中心、规则库、协作流
- P2：Hermes深接、海外扩展

建议文案：

- “先补强可用性，再扩展平台化能力。”

## 3. 每页内容写作规则（执行标准）

### 3.1 标题规则

- 标题必须是“结论句”，不能是空泛目录词
- 建议句式：
  - “我们解决的不是X，而是Y”
  - “A与B共用一个处置出口”
  - “当前版本已具备路演级可运行能力”

### 3.2 正文规则

- 每页正文控制在 35-55 字
- 卡片正文单条不超过 42 字
- 禁止出现大段技术名词堆叠

### 3.3 可信口径规则

- “已实现”只能写文档中明确实测通过的能力
- “规划中”必须显式标注，不可混写成“已具备”
- 避免绝对化词汇：
  - 禁用：100%自动判定、自动维权闭环、司法可直接采信
  - 推荐：疑似风险识别、建议动作、人工复核

### 3.4 配图规则

- 每个结构图页至少有 1 个核心图
- 每个真实界面页至少有 2 张截图
- 截图必须带标签（“前端主站”“工作台”“FastAPI Docs”）
